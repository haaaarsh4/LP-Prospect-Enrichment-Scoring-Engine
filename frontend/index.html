About & Design
Design decisions, tradeoffs, and what comes next
Scoring Architecture
Composite Score composite = sector_fit×0.35 + relationship_depth×0.30 + halo_value×0.20 + emerging_fit×0.15
35%
Sector Fit
True LP with private credit allocation AND a sustainability mandate. GPs and service providers score 1–3 regardless of ESG branding.
30%
Relationship Depth
Pre-computed from CRM data, used as-is. A warm relationship can lift a moderate prospect above a cold best-fit.
20%
Halo Value
Would this commitment signal credibility and attract other LPs? Brand recognition and institutional validation power.
15%
Emerging Fit
Structural appetite for Fund I/II managers. Dedicated programs and documented history of backing new GPs.
Key Design Decisions
01
Groq + LLaMA 3.3 70B for Enrichment
Cost-aware
Groq's free tier provides 100K tokens per day with excellent latency. LLaMA 3.3 70B produces reliable structured JSON when prompted carefully and has strong training coverage of institutional investors. The scoring logic lives entirely in the prompt — swapping providers is one line of code.
02
Org-Level Deduplication & Caching
Scalability
One enrichment call per organization, shared across all contacts from that org. The 99-contact sample collapses to ~94 unique orgs immediately. At 1,000 contacts where the same 200 firms appear repeatedly, savings compound dramatically. Results are cached in SQLite so re-runs skip already-processed orgs entirely.
03
Calibrated Org-Type Fallback Defaults
Scoring accuracy
When quota runs out or a parse fails, blanket 5.0 defaults add noise without signal. The system applies org-type-specific defaults instead: a Foundation defaults to sector=6.5, an RIA to sector=2.0. Meridian Capital Group gets a 2 on fallback, not a 5 — directionally correct even in failure.
04
Rescue Prompt Pattern
Reliability
The main enrichment prompt occasionally returns malformed JSON. Rather than immediately triggering a fallback, a stripped-down rescue prompt retries with half the token budget, requesting only the structured scores. This eliminates most silent failures without burning a full quota slot on a wasted call.
05
Token Efficiency Engineering
Cost-aware
Trimmed max_tokens from 1024 to 700. Shortened prompts ~40% by removing redundant instructions. Reduced per-request sleep from 4s to 3s while staying under Groq's 30 rpm cap. Net result: ~70% more successful enrichment calls within the daily free-tier quota.
06
SQLite with Full Upsert & Run Logging
Persistence
Zero configuration, zero dependencies beyond Python's standard library, sufficient for tens of thousands of records. The run_log table tracks every execution with token counts, cost estimates, and timestamps. Migrating to PostgreSQL when scale demands it requires changing one connection string.
07
Vanilla JS SPA, No Build Step
Simplicity
No React, no Webpack, no node_modules. One HTML file served by Flask. Any team member can run this on any machine with Python installed. Dashboard, prospect table, modal detail view, and upload flow all in a single file — deployment, debugging, and iteration are dramatically faster than a compiled frontend.
Honest Tradeoffs
API Tier
Free vs Paid
What We Gained
Zero cost prototyping. 100K daily tokens handles the 99-contact sample comfortably. Groq's latency is excellent for a demo.
The Limitation
At 1,000 contacts with 900 unique orgs, the daily limit runs out mid-run. A paid tier is needed for real scale — but the architecture makes provider-swapping trivial.
Data Source
Training vs live web
What We Gained
Simplicity and speed. No external API keys for search providers, no scraping complexity. The model knows a lot about institutional investors from training.
The Limitation
Scores reflect training-data snapshots. A fund that launched an emerging manager program last year may score low. Live search (Tavily, Brave) is the right long-term fix.
Model Design
Single vs pipeline
What We Gained
One prompt, one call, low latency. Easy to debug and iterate. Calibration anchors do a lot of work that a separate scoring model would otherwise need.
The Limitation
Enrichment and scoring are correlated — if the model misunderstands an org, both the summary and scores are wrong. Separating extraction from scoring improves robustness.
Processing
Sync vs async
What We Gained
Simple, predictable, easy to debug. Retry and backoff logic is easy to reason about when everything runs sequentially.
The Limitation
Processing 1,000 orgs sequentially with a 3s sleep takes ~50 minutes. Async batching with a token-bucket rate limiter would cut that by 70% or more.
Database
SQLite vs PostgreSQL
What We Gained
Zero setup, ships with Python, works anywhere. A single file you can copy, inspect with any SQL tool, and share. Perfect for a prototype.
The Limitation
Concurrent writes from multiple pipeline threads can cause locking. PostgreSQL with connection pooling is the right next step for multi-user or async batch deployments.
What I Would Improve With More Time
Live Web Search Integration
Connect Tavily or Brave Search so enrichment pulls current press releases, LP announcements, and recent fund commitments rather than relying on model training data alone. This would dramatically improve accuracy for newer or smaller organizations underrepresented in LLM training corpora.
Async Parallel Processing
Replace the sequential enrichment loop with an async token-bucket rate limiter processing 5–10 orgs concurrently. Combined with queue management and resume-on-failure, this brings 1,000 orgs from ~50 minutes down to under 10.
Dedicated GP/LP Classification Pass
A fast, lightweight first step that explicitly labels each org as LP, GP, hybrid, or service provider before scoring. This catches misclassifications early and prevents the main prompt from wasting tokens on entities that should automatically score 1–2.
Fundraiser Feedback Loop
Let fundraising team members flag incorrect scores directly in the UI. Those corrections feed back as few-shot examples on the next run, improving accuracy over time and creating a scoring layer proprietary to PaceZero's perspective.
Confidence-Gated Review Queue
When a dimension returns low confidence with limited evidence, route that prospect to a human review queue rather than publishing the score directly. Currently, low-confidence scores look identical to high-confidence ones — that's misleading for outreach prioritization.
Sector & Mandate Fit 35%
Does this entity's investment mandate align with PaceZero's sustainability-focused private credit strategy? An ideal prospect allocates capital into funds and has both a credit/lending allocation AND a sustainability/impact mandate. GPs, lenders, and service providers score very low.
Halo & Strategic Value 20%
Would winning this LP send a strong signal that attracts other LPs to PaceZero? High-halo LPs have strong brand recognition, institutional credibility, or are viewed as validators by other allocators in the market.
Emerging Manager Fit 15%
Does this LP show structural appetite for backing a Fund I/II or otherwise emerging manager? Looks for dedicated emerging manager programs, first-time fund mandates, or track record of backing new GPs.
Relationship Depth 30%
Pre-computed from CRM data. Derived from lifecycle stage, notes count, deal associations, and recency of contact. Reflects how warm the existing relationship is — not enriched by AI.
Estimated Check Size
Estimated commitment range based on AUM × org-type allocation %. Pension/Insurance 0.5–2%, Endowment/Foundation 1–3%, FoF/MFO 2–5%, SFO/HNWI 3–10%, Asset Manager/RIA 0.5–3%. Shown as low–high range. Blank when AUM is unknown.
